## P0 · 必须修复（运行时风险）

### P0-1 `MeetingViewModel.kt:406-408` 协程泄漏

**问题：** `endMeeting()` 中嵌套 `viewModelScope.launch`，导致 `transcriptionUseCase.stop()` fire-and-forget

```kotlin
// 修复前 (L392-409)
fun endMeeting() {
    viewModelScope.launch {
        ...
        viewModelScope.launch {       // ← 删除内层
            transcriptionUseCase.stop()
        }
        ...
    }
}

// 修复后
fun endMeeting() {
    viewModelScope.launch {
        ...
        transcriptionUseCase.stop()   // ← 直接调用
        ...
    }
}
```

**改动：** 删除 L406-408 的 `viewModelScope.launch { }` 包裹，直接调用 `transcriptionUseCase.stop()`

---

### P0-2 `FunAsrCloudEngine.kt:249-272` 裸 Thread 重连

**问题：** `scheduleReconnect()` 用 `Thread { Thread.sleep(); ... }.start()` 创建不受管理的线程

```kotlin
// 修复前
private fun scheduleReconnect() {
    ...
    Thread {
        try {
            Thread.sleep(delay)
            doConnect(url, config)
        } catch (e: InterruptedException) { ... }
    }.start()
}

// 修复后：使用协程
private fun scheduleReconnect() {
    ...
    reconnectJob = scope?.launch {
        delay(delay)
        doConnect(url, config)
    }
}
```

**改动：** 
- 新增 `private var scope: CoroutineScope?` 字段
- `start()` 中初始化 `scope = CoroutineScope(Dispatchers.IO + SupervisorJob())`
- `dispose()` 中 `scope?.cancel()`
- `scheduleReconnect()` 用 `scope.launch { delay(...); doConnect(...) }` 替代 Thread

---

### P0-3 `FunAsrCloudEngine.kt:169` reconnectAttempts 缺少 @Volatile

**问题：** 变量在 OkHttp callback 线程 + coroutine IO 线程间共享，无同步

```kotlin
// 修复前
private var reconnectAttempts = 0

// 修复后
@Volatile private var reconnectAttempts = 0
```

**改动：** 加 `@Volatile` 注解

## P1 · 应该修复（逻辑错误）

### P1-1 `EngineRouter.kt:73-79` 本地引擎硬禁用 vs. 模型已打包

**问题：** 
- 引擎路由器永远拒绝 `FUNASR_LOCAL`
- 但 `MeetingApplication` 中创建了 `FunAsrEngine` 实例
- `assets/models/sense-voice-small-cn.onnx`（232MB）仍打包在 APK
- 资源浪费 ~232MB

**方案 A（推荐）：** 改为运行时检测设备兼容性
```kotlin
// EngineRouter.kt
preferred == AsrEngineType.FUNASR_LOCAL -> {
    if (!FunAsrEngine.isDeviceSupported()) {
        throw NoEngineException("...")
    }
    ensureInitializedOrThrow(context, funAsrEngine!!)
    funAsrEngine!!
}
```
同时在 `FunAsrEngine` 加 `companion object { fun isDeviceSupported(): Boolean }`

**方案 B（最小改动）：** 仅加注释说明原因
```kotlin
// FIXME: sherpa-onnx OnlineRecognizer 在部分设备崩溃，待 root cause 确定后重新启用
// 相关：FunAsrEngine.kt 和 assets/models/ 暂时保留
```

**推荐 A**，改动量小，但恢复了一个被禁用的有效功能。

**问题：** FunASR 云端是默认引擎，但此方法不检查其 URL

```kotlin
// 修复前
fun hasAnyCloudAsrKey(): Boolean = hasTingwuKeys() || hasVolcengineKeys()

// 修复后
fun hasAnyCloudAsrKey(): Boolean =
```

**改动：** 补上 `hasFunAsrCloudUrl()` 检查

## P2 · 建议修复（代码质量）

### P2-1 `FunAsrEngine.kt:264-274` tmp 文件异常时残留

**问题：** `copyTo` 如果抛异常，`.tmp` 文件不清理

```kotlin
// 修复前
try {
    context.assets.open("$ASSET_MODEL_PATH/$MODEL_FILE_NAME").use { input ->
        tmpFile.outputStream().use { output -> input.copyTo(output) }
    }
    if (!tmpFile.renameTo(modelFile)) {
        tmpFile.copyTo(modelFile, overwrite = true)
        tmpFile.delete()
    }
} catch (e: Exception) {
    tmpFile.delete()    // 只删了 tmp，没处理半成品 modelFile
    throw e
}

// 修复后
try {
    context.assets.open("...").use { input ->
        tmpFile.outputStream().use { output -> input.copyTo(output) }
    }
    if (!tmpFile.renameTo(modelFile)) {
        tmpFile.copyTo(modelFile, overwrite = true)
        tmpFile.delete()
    }
} catch (e: Exception) {
    tmpFile.delete()          // 清理 tmp
    modelFile.delete()        // 清理可能的半成品
    throw e
}
```

**改动：** catch 块中加 `modelFile.delete()`
