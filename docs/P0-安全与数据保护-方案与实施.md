## 一、方案概览

MeetingTranscriber v0.85 在安全与数据保护方面存在三个关键缺失：
1. 本地存储完全明文（Room 数据库 + WAV 音频文件）
2. 异常崩溃无状态持久化（进程被杀后无法恢复会议）
3. 网络中断缓冲仅 20 秒且不持久化

第一轮 P0 针对这三项进行全面加固。

## 二、加密体系架构

```
AndroidKeyStore → Master Key (AES-256-GCM, KEK)
                       │
                  ┌─────┴──────┐
                  │ 加密 DEK   │
                  └─────┬──────┘
                        │
                  EncryptedSharedPreferences
                        │
                        ▼
                  DEK (AES-256)
                        │
          ┌─────────────┴─────────────┐
          │                           │
   getDatabasePassphrase()    getFileSecretKey()
   → hex(DEK) as ByteArray    → SecretKey
   → SQLCipher SupportFactory → EncryptedFile (Tink Streaming AEAD)
```

### 密钥生命周期

| 阶段 | 操作 |
|------|------|
| 首次启动 | AndroidKeyStore 生成 KEK → 生成随机 DEK → KEK 加密 DEK → 存入 EncryptedSharedPreferences |
| 后续启动 | 从 KeyStore 加载 KEK → 从 EncryptedSharedPreferences 读取加密 DEK → KEK 解密得到 DEK → 缓存在内存 |
| Debug 模式 | `CryptoManager.isEncryptionEnabled()` 返回 false，所有加密降级为明文 |
| 密钥丢失 | KeyStore/EncryptedSharedPreferences 任一损坏 → 降级为明文模式 + 日志告警 |

## 三、功能 1：本地存储加密

### 3.1 Room 数据库加密

**依赖**: `net.zetetic:android-database-sqlcipher:4.5.4`

**方案**：Room 构建时注入 `SupportFactory(CryptoManager.getDatabasePassphrase())`，SQLCipher 接管 SQLite 的页级加密。

**明文→加密迁移流程**：
1. 检测旧 DB 文件是否存在且可用明文 SQLiteDatabase 打开
2. 备份为 `.db.backup`
3. 逐行读取所有数据（meetings + transcript_segments）
4. 删除明文 DB 及 journal/WAL 文件
5. 通过 Room + SupportFactory 构建加密 DB
6. 通过 Room DAO 写回全部数据
7. 验证成功后删除备份
8. 任何步骤失败 → 从备份回滚

**DB 版本**: 2 → 3（新增 `recovery_state` 表）

### 3.2 WAV 文件加密

**依赖**: `androidx.security:security-crypto:1.1.0-alpha06` 中的 `EncryptedFile`

**自定义格式 MTEW**（避免 WAV 的 seek 回写需求）：
```
字节 0-3:   "MTEW" 魔数
字节 4-7:   版本号 (Int32 LE, =1)
字节 8-11:  采样率 (Int32 LE, =16000)
字节 12-13: 位深度 (Int16 LE, =16)
字节 14-15: 声道数 (Int16 LE, =1)
字节 16-19: 保留 (0x00000000)
字节 20+:   EncryptedFile 加密的 PCM 数据 (Tink AES-256-GCM-HKDF-4KB)
```

**WavRecorder 改动**：
- 构造函数接受可选 `SecretKey?`（null → 明文 WAV，非 null → 加密 MTEW）
- EncryptedFile 打开 OutputStream 后全程保持连接写入
- 加密失败自动降级为明文模式

**WavPlayer 改动**：
- 读取文件头 4 字节检测格式
- `RIFF` → 标准 WAV 播放（跳过 44 字节头）
- `MTEW` 或未知 → EncryptedFile 解密播放（跳过 20 字节头）
- 兼容旧明文文件

## 四、功能 2：异常崩溃数据保护

### 4.1 定时保存机制

**触发**: 会议开始后每 5 秒执行 `saveRecoveryState()`

**保存内容** (`recovery_state` 表)：

| 字段 | 说明 |
|------|------|
| meetingId | 主键，关联 meetings 表 |
| title | 会议标题 |
| startTime | 会议开始时间戳 |
| isOfflineMode | 是否离线模式 |
| isDemoMode | 是否演示模式 |
| currentTaskId | ASR 任务 ID（在线模式） |
| audioBufferFilePath | 溢出音频临时文件路径 |
| audioBufferFrameCount | 内存缓冲中的帧数 |
| speakerLabelMapJson | 说话人标签映射（JSON） |
| pendingSegmentsJson | 尚未写入 DB 的转写片段（JSON） |
| lastSavedAt | 最后保存时间戳 |

### 4.2 崩溃检测与恢复

```
应用启动
  │
  ├─ MeetingApplication.onCreate()
  │    └─ checkRecoveryState()
  │         ├─ 查 recovery_state 表最新记录
  │         ├─ 超过 24h → 清除
  │         └─ 未超过 → 设 pendingRecoveryState
  │
  ├─ MainActivity.onCreate()
  │    └─ pendingRecoveryState != null?
  │         ├─ 是 → AlertDialog (恢复/放弃)
  │         └─ 否 → 正常启动
  │
  └─ 用户选择"恢复"
       └─ MeetingFragment.recoverFromCrash(state)
            └─ MeetingViewModel.recoverFromCrash()
                 ├─ 恢复 meetingId / taskId / startTime
                 ├─ 恢复 speakerLabelMap (JSON 反序列化)
                 ├─ 恢复 pendingSegments → 写入 DB
                 ├─ 重新启动音频采集
                 ├─ 重新连接 ASR WebSocket
                 ├─ 回放溢出音频文件
                 └─ 启动定时保存
```

### 4.3 最后防线

ViewModel `onCleared()` 中使用 `runBlocking { withTimeout(2000) { saveRecoveryState() } }` 做崩溃前的最后一次保存。

## 五、功能 3：断点续传完善

### 5.1 缓冲扩容

| 参数 | 旧值 | 新值 |
|------|------|------|
| MAX_BUFFER_SIZE | 200 帧 (20s) | 1200 帧 (120s) |

### 5.2 内存→磁盘溢出

当 `ConcurrentLinkedQueue` 达到上限后：
1. 创建临时文件 `cacheDir/audio_overflow_{timestamp}.pcm`
2. 后续音频帧写入该文件
3. `flushBuffer()` 重连后先清空内存缓冲，再流式回放溢出文件，最后删除

### 5.3 恢复集成

- `saveRecoveryState()` 同步记录溢出文件路径和帧数
- `recoverFromCrash()` 中重建 ASR 连接后回放溢出音频
- 正常结束会议时清理溢出文件
